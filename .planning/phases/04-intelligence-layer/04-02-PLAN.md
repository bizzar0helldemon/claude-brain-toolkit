**Part A: Add two functions to `hooks/lib/brain-path.sh`** (append after the existing `emit_json` function):

1. `init_pattern_store` — Takes one arg: store path. Creates `$BRAIN_PATH/brain-mode/` dir if needed. If the store file does not exist, creates it with empty schema `{"version":"1","created":"$NOW","updated":"$NOW","patterns":[]}` via atomic temp+mv. If it already exists, no-op. Returns 0.

2. `update_encounter_count` — Takes two args: store path, error message. Uses jq to: set `.updated` to now, iterate `.patterns[]`, for any pattern where the error (ascii_downcase) contains the pattern key (ascii_downcase), increment `.encounter_count` by 1 and set `.last_seen` to now. Writes via atomic temp+mv. On write failure, logs via `brain_log_error "PatternStore" "atomic write failed"`. Returns 0 regardless (never crash the hook over a count update).

Use `date -u +"%Y-%m-%dT%H:%M:%SZ"` for timestamps (already used elsewhere in brain-path.sh).

**Part B: Extend `hooks/post-tool-use-failure.sh`** to add pattern matching after the existing error logging:

0. **Fix existing graceful degradation bug:** Change the existing `exit 1` after `brain_path_validate` failure to `exit 0`. PostToolUseFailure cannot block tool use, so the hook must exit 0 on all paths. The current `exit 1` is a Phase 1 oversight that would cause hook failures when BRAIN_PATH is unset.

1. After the existing `brain_log_error` call, define `PATTERN_STORE="${BRAIN_PATH}/brain-mode/pattern-store.json"`
2. If the file does not exist, emit the existing simple JSON response and exit 0 (same as current behavior)
3. If the file exists, use jq to find a matching pattern: iterate `.patterns[]`, select where `$ERROR_MSG | ascii_downcase | contains(.key | ascii_downcase)` OR `$COMMAND | ascii_downcase | contains(.key | ascii_downcase)`. Extract `.solution` from the first match. Pipe through `head -1` to get only the first match.
4. If a match is found:
   - Call `update_encounter_count "$PATTERN_STORE" "$ERROR_MSG"`
   - Build output JSON with `additionalContext` containing the past solution text: `{"hookSpecificOutput":{"hookEventName":"PostToolUseFailure","additionalContext":"Past solution found for this error:\n\n$MATCH"}}`
   - Emit via `emit_json` and exit 0
5. If no match, emit the existing simple response (same as current behavior)

Also extract COMMAND from hook input (add alongside existing TOOL_NAME and ERROR_MSG extraction):
`COMMAND=$(printf '%s' "$HOOK_INPUT" | jq -r '.tool_input.command // ""')`

IMPORTANT: The hook must remain safe for fresh installs where no pattern-store.json exists (early return before any jq pattern read). Wrap the jq match in `2>/dev/null` to handle malformed JSON gracefully.
  </action>

   - **Error key**: A short, distinctive substring that appears in the error message (e.g., "invalid JSON", "ECONNREFUSED", "permission denied"). This is what the hook matches against.
   - **Solution**: The fix or workaround in markdown (1-3 sentences). This is what gets injected into Claude's context when the error recurs.
   - **Pattern ID**: A kebab-case identifier (e.g., "jq-invalid-json"). Auto-generated from the key if not provided.

Do not source brain-path.sh for initialization — use the Write tool directly (more reliable than sourcing shell functions at runtime).
3. Check if a pattern with the same key already exists — if so, ask the user whether to update the solution or skip

   ```
   {id, key, solution, source_file: null, encounter_count: 0, first_seen: NOW, last_seen: NOW}
   ```
5. Confirm: "Pattern added: [key] — will match future errors containing this text and surface the solution automatically."
