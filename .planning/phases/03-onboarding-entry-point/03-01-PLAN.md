- `tools: Read, Write, Edit, Bash, Grep, Glob, Agent`

Do NOT include a `skills:` frontmatter field. Research question 3 (open question about whether skills preload when agent IS the main thread via --agent flag) is unresolved. The skills (brain-capture, daily-note, brain-audit) are already installed to `~/.claude/skills/` and available via slash commands — they don't need frontmatter preloading to function. System prompt content must include these sections:

1. **Identity** — "You are Claude in brain mode — a knowledge-aware assistant that actively manages a personal vault." One paragraph explaining the role.

2. **Vault Location** — Instruct Claude to read BRAIN_PATH from the environment. All brain operations read/write to this directory.

3. **Session Start Behavior** — This section must handle THREE distinct startup states:

   **(a) No hook output at all (hooks not registered):** If no brain context summary was injected by the SessionStart hook (i.e., there is no additionalContext mentioning brain/vault at all), this means the hooks are not deployed yet. The user needs to run the setup script first. Claude must tell the user: "Brain hooks are not installed yet. Run `bash onboarding-kit/setup.sh` from the claude-brain-toolkit directory to deploy hooks, then restart Claude Code." This is a distinct case from degraded:true — it means SessionStart never fired at all.

   **(b) Degraded context (degraded:true):** When the session starts and the injected brain context contains `"degraded": true` or `"error":"BRAIN_PATH is not set"`, Claude must immediately tell the user about the issue and offer to run `/brain-setup`. This means hooks ARE installed but BRAIN_PATH is not configured.

   **(c) Normal context:** If brain context loaded successfully, acknowledge it briefly and proceed.

   The system prompt must clearly distinguish these three cases and specify the correct action for each.

4. **Proactive Knowledge Capture** — After significant work (commits, completing features, solving hard bugs): offer to run `/brain-capture`. At session end or before `/clear`: the Stop hook handles this automatically via decision:block, but Claude should mention what was captured. When the user asks about past work: consult the vault context injected at session start.

5. **When Vault Is Empty** — If brain context loaded but shows 0 project entries and 0 pitfalls, suggest running `/brain-scan` to catalog the current project into the vault. Don't push — just mention it once.

6. **Available Skills** — List the brain skills Claude can use: `/brain-capture` (extract patterns), `/daily-note` (journal entry), `/brain-audit` (vault health check), `/brain-scan` (catalog a project), `/brain-setup` (first-time onboarding).

Keep the system prompt concise — under 800 words. This content is injected into every session; bloat costs tokens.

Create `onboarding-kit/skills/brain-setup/SKILL.md` — the interactive onboarding wizard skill.

Frontmatter:
- `name: brain-setup`
- `description: First-time onboarding wizard for Claude Brain. Guides the user through creating a vault directory and configuring BRAIN_PATH. Run this when BRAIN_PATH is unset or the vault directory is missing.`

The skill body must handle two distinct cases:

**Case A: BRAIN_PATH unset (new user)**
Triggered when session context contains `"error":"BRAIN_PATH is not set"` or when user runs `/brain-setup` manually with no BRAIN_PATH set.

Flow:
1. Ask user where they want their brain vault. Provide examples: `~/Documents/claude-brain`, `~/brain`, `~/Desktop/claude-brain`. Explain briefly what this directory is for.
2. When user provides path:
   a. Expand `~` to `$HOME` using bash
   b. Create directory: `mkdir -p "<path>"`
   c. Write BRAIN_PATH to `~/.claude/settings.json` env block using jq (see below)
   d. Write BRAIN_PATH to shell profile (see below)
   e. Show confirmation with the path
   f. Instruct user to restart Claude Code: "Please restart Claude Code (`/exit` then `claude`) so the new BRAIN_PATH takes effect. After restarting, brain mode will load vault context automatically."

**Case B: BRAIN_PATH set but directory missing**
Triggered when session context contains `"offer_create": true`.

Flow:
1. Tell the user: "Your BRAIN_PATH is set to [path] but that directory doesn't exist."
2. Offer two choices:
   - "Create the directory at the current path" — run `mkdir -p "$BRAIN_PATH"`, then continue (no restart needed since BRAIN_PATH is already configured)
   - "Update BRAIN_PATH to a new location" — follow the same flow as Case A

**settings.json env update pattern (include in skill body as bash code block):**
```bash
SETTINGS="$HOME/.claude/settings.json"
if [ ! -f "$SETTINGS" ]; then
  printf '{}\\n' > "$SETTINGS"
fi
jq --arg p "$BRAIN_PATH_VALUE" '.env.BRAIN_PATH = $p' "$SETTINGS" > "$SETTINGS.tmp" && mv "$SETTINGS.tmp" "$SETTINGS"
```

**Shell profile update pattern (include in skill body as bash code block):**
```bash
if [ -f "$HOME/.zshrc" ]; then
  PROFILE="$HOME/.zshrc"
elif [ -f "$HOME/.bash_profile" ]; then
  PROFILE="$HOME/.bash_profile"
elif [ -f "$HOME/.bashrc" ]; then
  PROFILE="$HOME/.bashrc"
else
  PROFILE="$HOME/.bashrc"
fi

if ! grep -q 'BRAIN_PATH' "$PROFILE" 2>/dev/null; then
  printf '\\n# Claude Brain vault path\\nexport BRAIN_PATH="%s"\\n' "$BRAIN_PATH_VALUE" >> "$PROFILE"
fi
```

**jq fallback:** Add a note in the skill: if `jq` is not available on PATH, fall back to using the Write tool to directly edit `~/.claude/settings.json` as JSON text. The skill runs inside Claude Code where Claude has file editing tools — jq is preferred but not the only option.

**Important pitfall to address in the skill text:** On Windows (Git Bash), shell profile files may not exist. The settings.json env block is the primary reliable channel on Windows. The shell profile write is secondary. Document this clearly: "Note: On Windows with Git Bash, the settings.json env block is the primary way BRAIN_PATH is delivered to hooks. The shell profile export is a convenience for interactive terminal use."

Keep the skill under 600 words — it's a guided conversation, not a manual.
